# 需求质量检查清单

## 八大质量标准

| # | 标准 | 检查要点 | 反面案例 |
|---|------|----------|----------|
| 1 | **完整性** | 所有场景、逆向流程、异常处理是否覆盖？隐藏干系人是否遗漏？外部依赖是否梳理？ | 只做了发优惠券忘了引流环节 |
| 2 | **正确性** | 逻辑是否符合业务规则？审批流程是否正确？ | 三种假期审批逻辑写混 |
| 3 | **精确性** | 有无模糊词汇？边界用数学符号表达？控件是否精确指代？ | "高亮"—没指定RGB；"之后"—没说明是否包含当天 |
| 4 | **一致性** | 术语、格式、工具是否统一？英文命名是否规范？ | 同一概念用了三个不同名字 |
| 5 | **复用性** | 是否有重复定义？消息列表、术语定义、通用控件是否集中管理？ | 每次弹出提示都重新写消息文本 |
| 6 | **易读性** | 长句子是否拆为表格？缩进条件是否改为矩阵？流程图是否按阶段拆分？ | "当当当当"式缩进条件嵌套 |
| 7 | **可测试性** | 每个功能是否有明确验收条件？测试环境和数据是否可指定？ | "支撑1万人同时在线"—没说明测试条件 |
| 8 | **灵活性** | 新增分类是加列还是重构？修改规则是否影响架构？ | 会员等级增加一级就要全部重写 |

## 常见错误检查清单 (44项精选)

### 模糊表达 (12项)
- [ ] "高亮" → 指定RGB色值
- [ ] "模糊查询" → 明确类型（前半/后半/包含）
- [ ] "之后/之前" → 用数学符号 `>=` `<=`
- [ ] "超过/之内" → 用开闭区间 `[]` `()`
- [ ] 控件无名 → 编号指代
- [ ] 长句子 → 拆为表格
- [ ] "XX中/待XX" → 用过去式
- [ ] 顺序/倒序 → 改用升序/降序
- [ ] 本周/本月 → 明确定义
- [ ] 编号规则 → 表格说明每部分

### 需求遗漏 (15项)
- [ ] 场景遗漏（没有用闭环思维）
- [ ] 逆向流程遗漏（取消/退货/退款）
- [ ] 状态-操作组合未100%覆盖
- [ ] 角色遗漏（财务/风控/老板/顾问）
- [ ] 隐藏干系人
- [ ] 小功能遗漏（个性化设置）
- [ ] 整块功能遗漏（引流/合同处理）
- [ ] UI细节（数字格式、加载状态、空白选项）
- [ ] 外部依赖系统遗漏
- [ ] 校验遗漏 → 用校验矩阵
- [ ] 权限遗漏（角色/组织/数据/状态）
- [ ] 多样性遗漏（设备/角色/业务种类/操作组合）

### 其他问题
- [ ] 动画/交互细节未描述（方向、角度、时长、缓动）
- [ ] 响应式规则未定义
- [ ] 紧急预案缺失（紧急下架）
- [ ] 竞合场景解救措施缺失

## 不切实际的假设 (20条)
1. 网络一直畅通
2. 访问的人数很少
3. 开发一定可以实现
4. 不会有恶意用户
5. 用户的输入不会包含特殊字符
6. 所有用户都会按规则操作
7. 没有用户从国外访问
8. 所有设备字符集都一样
9. 没有用户使用旧设备
10. 所有用户都用浏览器访问
11. 活动一直都很正常
12. 测试过了质量就有保了
13. 用户无法查看源代码
14. 前端校验了就可以了
15. 文件放服务器上很安全
16. 有用户名密码就能识别身份
17. 一个人能正常使用所有人都可以
18. 仿照别人设计就一定能吸引用户
19. 功能越多越好
20. 过去一直好→未来一定好
