## create-order-pad.html 多张金额卡抵扣：应用逻辑说明与校验（评审用）

### 1. 目标与范围

本说明用于对齐并校验 `create-order-pad.html` 中"金额卡应用/抵扣"的核心业务逻辑与边界处理，确保文档描述与页面实现一致。

本说明覆盖（以当前实现为准）：

- 金额卡数据结构（商品行绑定、弹窗状态、明细行渲染）
- 选卡/替换/删除/防重复、默认划扣生成、0 金额自动移除
- 多张金额卡组合使用的计算规则（逐张应用、折扣率独立计算）
- 划扣金额归一化（跨行、跨卡余额递减）、数量/折扣/折后价变化时的联动刷新策略
- 整数运算精度保证（使用分进行计算，避免浮点精度问题）
- 订单汇总的金额卡统计口径（金额卡支付/待收款）
- 支付页 `payment.html` 的金额卡跨页面展示与"剩余应支付"口径（仅含金额卡相关部分）
- 整单改价弹窗与金额卡的联动逻辑（最低改价金额限制）

不在本说明覆盖（当前原型未实现或仅模拟）：

- 金额卡有效性（过期/冻结/不可用）与后端校验的真实拦截
- 真实交易扣减、并发一致性、退款/撤销后的回滚与审计流水
- 支付页的完整财务链路（找零、退款、跨渠道对账等）。`payment.html` 仅对"金额卡抵扣展示、剩余应支付口径、可见性规则"做前端模拟

### 2. 术语与数据口径

- **原价**：商品行展示的单价（`.col-original`，单位元）
- **购买数量**：商品行数量输入（服务/产品为 `data-field="buy"`；疗程为 `data-field="buyQty"`）
- **折扣**：商品行折扣输入（百分比 0–100，整数）
- **折前价值**：商品行原价 × 购买数量（元）
- **金额卡折扣（折）**：金额卡在不同商品类型下的折扣（例如 7.5 折 = 0.75）
- **划扣金额**：某张金额卡在当前商品行上用于抵扣的金额（元）
- **划扣原价值**：划扣金额 ÷ 该金额卡在对应商品类型的折扣率（元），即该划扣金额对应的商品原价
- **行应付金额（本品剩余应付）**：`折前价值 - Σ(该行所有金额卡划扣原价值之和)`，并做 0 下限保护
- **实际支付金额**：`Σ(该行所有金额卡划扣金额) + 行应付金额`（元）
- **商品折扣率**：`实际支付金额 ÷ 折前价值 × 100`（整数百分比）
- **订单金额卡支付**：当前订单内所有商品行的金额卡划扣金额之和
- **订单待收款**：`整单应收（含整单改价后的 displayFinal）- 订单金额卡支付`，并做 0 下限保护
- **支付页待收款（payment.html 展示基数）**：开单页跳转时传入的 `payable`（两位小数）
- **支付页剩余应支付（payment.html）**：`支付页待收款 - 支付页金额卡抵扣总额`，并做 0 下限保护
- **整单改价最低金额**：整单改价弹窗中允许的最低应收金额，该金额受金额卡已划扣金额限制

### 3. 数据结构与渲染载体

#### 3.1 商品行与金额卡绑定（DOM 承载）

- `tr.item-row`：可应用金额卡的商品行（服务/产品/疗程）
- `data-moneycards`：JSON 数组，形如 `[{ "cardId": "mc_001", "amount": 100.00 }, ...]`
- 详情行：在 `item-row` 与其 `item-action-row` 之间插入 `tr.moneycard-detail-row`，用于展示已选金额卡明细（划扣金额只读展示）
- 选择按钮：商品行操作区按钮 `[data-action="open-moneycard-modal"]`，文案随"已选 X 张金额卡"变化；当行应付=0 时置灰不可继续选卡

#### 3.2 金额卡列表与弹窗状态

- 金额卡卡池：`MONEY_CARD_OPTIONS`（包含 `id/name/balance/discounts`；余额为前端模拟的初始余额）
- 选卡弹窗状态：`moneyCardModalState`
  - `row`：当前操作的商品行
  - `mode/editIndex`：新增 or 替换（编辑）
  - `pendingId`：弹窗内当前选中的金额卡 id（支持"不使用金额卡"空值）

### 4. 业务规则 → 技术落地逻辑（与实现一致）

#### 4.1 选卡/替换/删除/防重复

**规则 1：新增金额卡（Add）**

- 当在弹窗中选择某金额卡并确认：
  - 若该行已存在同 cardId 的金额卡，则阻止重复添加（无更新）
  - 为新增卡生成默认划扣金额（见 4.3 规则 1），若默认划扣 ≤ 0，则直接关闭弹窗且不新增

**规则 2：替换金额卡（Replace / Edit）**

- 在明细行点击"替换"进入编辑模式：
  - 若选择"不使用金额卡"（空值）并确认：仅移除该 index 对应的金额卡
  - 若选择其他金额卡并确认：
    - 不允许替换成该行已存在的其他金额卡（防重复）
    - 替换仅更新 cardId，随后触发归一化（会按余额/剩余应付对划扣金额进行裁剪）
  - 编辑完成后清除 `moneycardTouched`

**规则 3：删除金额卡**

- 明细行点击"删除"：移除对应金额卡并触发归一化；同时清除 `moneycardTouched`

#### 4.2 计算规则（核心）

**规则 1：整数运算保证精度**

- 所有金额计算使用\*\*分（fen）\*\*为单位进行运算，避免 JavaScript 浮点数精度问题
- 辅助函数：
  - `toFen(yuan)`：将元转换为分 `Math.round(yuan * 100)`
  - `toYuan(fen)`：将分转换为元 `Math.round(fen) / 100`
- 中间计算全部使用分（整数），最终显示时转回元

**规则 2：折前价值计算**

```
折前价值（分）= 原价（元）× 购买数量 × 100
```

**规则 3：逐张金额卡划扣计算（PRD规则 - 首次应用场景）**
对于每张已应用或待应用的金额卡，按应用顺序逐张计算：

```
划扣金额（分）= min(剩余折前价值（分）× 折扣率, 卡余额（分）)
划扣原价值（分）= 划扣金额（分）× 100 ÷ 折扣率（折扣率 = 折数字 ÷ 10）
剩余折前价值（分）= 剩余折前价值（分）- 划扣原价值（分）
```

其中：

- `折扣率` = 金额卡在该商品类型下的折扣值 ÷ 10（如 8 折 = 0.8）
- `卡余额`取该卡在当前订单中的运行中余额（跨行共享，按处理顺序递减）

**规则 4：行应付金额（剩余应付）- PRD规则**

```
行应付金额（分）= max(0, 折前价值（分）- 已应用金额卡划扣原价值之和（分）)
```

**规则 5：实际支付金额与折扣率**

```
实际支付金额（分）= 划扣金额之和（分）+ 行应付金额（分）
商品折扣率（%）= round(实际支付金额（分）÷ 折前价值（分） × 100)
```

**规则 6：跨行余额递减（同一张卡在订单内共享余额）**

- 归一化按固定行顺序遍历：服务区 → 产品区 → 疗程区（DOM 顺序）
- 对每张金额卡维护一个"运行中余额 runningBalance"：
  - 初始值为 `MONEY_CARD_OPTIONS.balance`（转分为单位）
  - 每处理到一行的某张卡：先取该卡"处理前余额 before"，再扣减本行的划扣 amount 得到 after，写回 runningBalance

**规则 7：0 金额保留（所有场景）**

- 任意场景下出现划扣金额 ≤ 0：**保留**该金额卡，不移除

**规则 8：新划扣规则（已应用卡后修改场景）**
当商品行已有金额卡且用户手动修改折扣/折后价后（`userDiscountOverride=1`），按以下规则计算：

```
商品折后价（分）= 商品原价（分）× 购买数量 × 折扣率（用户设置的折扣 ÷ 100）
划扣金额（分）= min(剩余应付金额（分）, 卡余额（分）)
剩余应付金额（分）= 剩余应付金额（分）- 划扣金额（分）
```

**规则 9：行应付金额（剩余应付）- 新规则**

```
行应付金额（分）= max(0, 商品折后价（分）- 已应用金额卡划扣金额之和（分）)
```

**规则 10：规则选择逻辑**

- 当 `data-moneycards` 为空（首次应用）：使用 PRD规则（规则3、规则4）
- 当 `data-moneycards` 非空且 `userDiscountOverride='1'`（用户手动修改过）：使用新规则（规则8、规则9）
- 当 `data-moneycards` 非空且 `userDiscountOverride!=='1'`（未手动修改）：使用 PRD规则（规则3、规则4）

#### 4.3 归一化的触发时机

任一触发会执行 `normalizeAndRenderMoneyCards()` 并联动更新行应付与订单汇总：

- 新增/替换/删除金额卡
- 数量变化（buy/buyQty）：300ms 防抖触发
- 折扣/折后价变化：raf/blur 定时触发（折后价 blur 后 80ms 内触发一次全量归一化）

#### 4.4 用户手动修改折扣/折后价与金额卡的联动逻辑

**规则 1：手动修改折扣（仅在用户确认提交后触发）**
- 当行已应用金额卡时，用户通过数字键盘确认提交折扣：
  - 计算新折后价：`新折后价 = 商品原价 × 购买数量 × 新折扣率`
  - 校验：若新折后价 > 当前折后价，阻止修改并弹出提示窗口："修改折后价必须大于0元并且小于等于原折扣价格(X元)，如需赠送请勾选【设为赠送权益】"
  - 校验通过后：
    - 标记 `userDiscountOverride=1`
    - **保留已应用的金额卡**
    - 按新规则（规则8）计算每张卡的划扣金额：`划扣金额 = min(剩余应付金额, 卡余额)`，其中剩余应付金额初始为新折后价，逐张递减
    - 计算剩余应付金额：`剩余应付 = max(0, 新折后价 - Σ划扣金额)`
    - 同步更新商品操作行应付金额

**规则 2：手动修改折后价（仅在用户确认提交后触发）**
- 当行已应用金额卡时，用户通过数字键盘确认提交折后价：
  - 校验：若新折后价 > 当前折后价 或 新折后价 ≤ 0，阻止修改并弹出提示窗口："修改折后价必须大于0元并且小于等于原折扣价格(X元)，如需赠送请勾选【设为赠送权益】"
  - 校验通过后：
    - 标记 `userDiscountOverride=1`
    - **保留已应用的金额卡**
    - 折扣按 `折扣% = 折后价 ÷ (原价 × 购买数量) × 100` 反算，保留 2 位小数
    - 按新规则（规则8）计算每张卡的划扣金额：`划扣金额 = min(剩余应付金额, 卡余额)`，其中剩余应付金额初始为新折后价，逐张递减
    - 计算剩余应付金额：`剩余应付 = max(0, 新折后价 - Σ划扣金额)`
    - 同步更新商品操作行应付金额

**规则 3：userDiscountOverride标记管理**
- 设置时机：用户手动修改折扣或折后价时设置为 `'1'`
- 清除时机：
  - 首次应用金额卡（`data-moneycards` 为空时应用第一张卡）
  - 删除所有金额卡后
  - 折扣重置为100%时同步清除
  - 勾选商品行上的"赠送"复选框时（`onGiftFlagToggle`方法触发）
- 保持时机：数量变化时不修改标记状态

**规则 4：首次应用金额卡前重置**
- 当 `data-moneycards` 为空时应用第一张金额卡：
  - 重置折扣为 100%
  - 重置折后价为 `原价 × 购买数量`
  - 清除 `userDiscountOverride` 标记
  - 按 PRD规则（规则3、规则4）计算划扣金额：
    - 划扣原价值 = 划扣金额 ÷ 金额卡折扣率
    - 剩余折前价值 = 剩余折前价值 - 划扣原价值
  - 按 PRD规则（规则4）计算商品操作行应付金额：`行应付金额 = max(0, 折前价值 - Σ划扣原价值)`
  - 同步更新商品操作行应付金额

**规则 5：删除所有金额卡后重置**
- 删除所有金额卡后：
  - 重置折扣为 100%
  - 重置折后价为 `原价 × 购买数量`
  - 重置应付金额为折后价
  - 清除 `userDiscountOverride` 标记

**规则 6：数量变化处理**
- 数量变化时不重置折扣和 `userDiscountOverride` 标记
- 根据当前 `userDiscountOverride` 状态选择计算规则（规则10）

#### 4.5 划扣金额只读（不可手动修改）

**规则 1：划扣金额展示为禁用状态**

- 已应用金额卡的划扣金额输入框 `.moneycard-deduct-input` 设置为 `disabled`
- 输入框视觉上置灰显示：`background-color: #f3f4f6; color: #6b7280; cursor: not-allowed;`
- 点击输入框不会触发数字键盘

**规则 2：划扣金额由系统自动计算**

- 划扣金额根据第 4.2 节的计算规则自动确定
- 用户无法手动调整划扣金额
- 替换或删除金额卡时会触发归一化，自动重新计算所有卡的划扣金额

#### 4.6 订单汇总统计口径（与金额卡抵扣一致）

- 订单金额卡支付：汇总所有 `tr.item-row[data-moneycards]` 内的 `amount` 之和（>0 的项）
- 待收款：`整单应收（含整单改价后的 displayFinal）- 订单金额卡支付`，并做 0 下限保护
- 仅当"订单金额卡支付 > 0"时，底部汇总区域展示独立字段「金额卡支付」

#### 4.7 整单改价弹窗与金额卡的联动规则

**规则 1：最低改价金额的计算**
整单改价弹窗中显示的"最低改价金额"受金额卡应用情况影响：

```
最低改价金额 = Σ(所有商品行已应用金额卡的划扣金额之和)
最低折扣 = (最低改价金额 ÷ 整单折后总价) × 10
```

**规则 2：金额卡影响最低改价金额的原理**

- 金额卡已划扣的金额是"锁定"的，无法通过整单改价降低
- 整单改价只能在金额卡划扣金额的基础上调整（只能调高，不能调低到划扣金额以下）
- 公式理解：`最终应收 ≥ 金额卡已划扣金额`，即 `整单改价金额 ≥ minAmount`

**规则 3：改价确认时的校验**
当用户点击"确定"确认整单改价时：

```
if (输入金额 < 最低改价金额) {
    显示错误提示
    阻止确认
    恢复输入框为原始值
}
```

**规则 4：改价对金额卡的影响**

- 整单改价不会影响已应用的金额卡（划扣金额保持不变）
- 改价只影响"待收款"部分，即顾客还需要支付的金额
- 改价后金额卡支付占比会相应变化

**规则 5：无金额卡时的最低改价金额**
当订单中没有应用任何金额卡时：

```
最低改价金额 = ¥0.00
最低折扣 = 0折
```

#### 4.8 支付页 payment.html：金额卡跨页面展示与剩余应支付口径

**规则 1：跨页面数据同步（开单页 → 支付页）**

- 开单页在点击「结账」跳转前，会汇总"按卡聚合"的金额卡抵扣数据并写入 localStorage：
  - `checkout_moneycards_${orderId}`：当前订单的金额卡汇总（优先读取）
  - `latest_checkout_moneycards`：兜底读取（用于缺少 orderId 命中时）
- 数据结构：
  - `cards`: `[{ cardId, name, amount }]`（amount 保留 2 位小数且 >0）
  - `total`: `Σ amount`（2 位小数）
  - `hasAvailableMoneyCards`: 是否存在可应用金额卡的卡池（原型字段，用于可见性策略扩展）

**规则 2：支付页金额口径**

- 支付页待收款（展示基数）：始终等于开单页传入的 `payable`（不因金额卡变化而改变）
- 支付页金额卡抵扣总额：读取 localStorage 后计算 `total`
- 支付页剩余应支付：`max(0, 待收款 - 抵扣总额)`，所有金额统一两位小数

**规则 3：防超额校验与裁剪（支付页）**

- 若"金额卡抵扣总额 > 待收款"，支付页会将抵扣总额裁剪为待收款，并提示"已自动裁剪"
- 同时会对展示明细做裁剪归一化（按金额从大到小分配，直到抵扣总额达到待收款）

**规则 4：可见性（支付页）**

- 若"无已应用金额卡"（抵扣总额=0）：支付页隐藏所有金额卡相关内容（包含金额卡抵扣行、明细列表、相关说明、以及剩余应支付行），仅展示待收款基数；页面不应出现空白占位
- 若"存在已应用金额卡"（抵扣总额>0）：按"待收款 - 金额卡抵扣 = 剩余应支付"的顺序展示三行核算信息，并展示金额卡抵扣明细与汇总

**规则 5：金额卡名称标识**

- 支付页 `moneycard-summary-name` 区域在金额卡名称后显示【本金】或【赠金】标识
- 判断逻辑：卡ID以 `__gift` 结尾为赠金，否则为本金

### 5. 边界与异常处理（实现已覆盖）

| 边界场景       | 处理方式                         |
| ---------- | ---------------------------- |
| 卡余额为 0     | 划扣金额 = 0，**保留该卡（不移除）**          |
| 折扣率为 0     | 跳过该卡，不参与计算                   |
| 剩余应付 ≤ 0   | 行应付 = 0，后续卡划扣 = 0，**保留所有金额卡**    |
| 多张卡均余额不足   | 按顺序逐张应用，每张卡尽可能抵扣             |
| 多张卡叠加覆盖    | 最后一张卡若余额充足则完全覆盖，剩余应付 = 0     |
| 不同折扣率（PRD规则） | 每张卡按各自折扣率独立计算划扣原价值           |
| 不同折扣率（新规则） | 不考虑金额卡折扣率，直接按商品折后价计算         |
| 数量变化       | 300ms 防抖触发全量归一化，**不重置折扣和标记**      |
| 整数运算精度     | 使用分（fen）计算，最后转回元，避免 0.01 元误差 |
| 整单改价低于最低金额 | 显示错误提示，阻止确认，恢复原始值            |
| 金额卡已应用后改价  | 改价不影响金额卡划扣金额，只影响待收款部分        |
| 修改折扣/折后价超出限制 | 弹出提示窗口阻止修改，提示："修改折后价必须大于0元并且小于等于原折扣价格(X元)，如需赠送请勾选【设为赠送权益】" |
| 首次应用金额卡前   | 自动重置折扣为100%，折后价为原价×数量，清除标记    |
| 删除所有金额卡后   | 自动重置折扣为100%，折后价为原价×数量，应付金额=折后价，清除标记 |
| 删除单张金额卡（非全部） | 不清除 userDiscountOverride 标记，保留用户修改状态   |

#### 示例4：浮点精度问题规避

**场景**：home类型，原价¥200，数量1

- 卡1：余额¥60，折扣9折
- 卡2：余额¥1000，折扣8折

使用整数运算（分）：

| 计算步骤   | 分计算                              | 转元后     |
| ------ | -------------------------------- | ------- |
| 折前价值   | 20000分                           | ¥200.00 |
| 卡1划扣   | min(20000×90÷100, 6000)=6000分    | ¥60.00  |
| 卡1原价值  | 6000×100÷90=6667分                | ¥66.67  |
| 剩余折前价值 | 20000-6667=13333分                | ¥133.33 |
| 卡2划扣   | min(13333×80÷100, 100000)=10667分 | ¥106.67 |
| 卡2原价值  | 10667×100÷80=13334分              | ¥133.34 |
| 总原价值   | 6667+13334=20001分                | ¥200.01 |

**问题**：总原价值多了1分（¥0.01）

**解决方案**：剩余应付使用 `max(0, 折前价值 - 总原价值)` 保护

```
剩余应付 = max(0, 20000 - 20001) = 0
实际支付 = 10667 + 0 = ¥106.67
```

最终结果：划扣¥166.67，剩余应付¥0.00，折扣率83%

#### 示例5：整单改价与金额卡的联动

**场景**：

- 订单包含两个商品行
- 商品A：原价¥500，应用金额卡X划扣¥100（8折）
- 商品B：原价¥300，应用金额卡Y划扣¥150（7折）
- 无金额卡时整单折后总价：¥500 + ¥300 = ¥800

**最低改价金额计算**：

```
最低改价金额 = ¥100 + ¥150 = ¥250
最低折扣 = (250 ÷ 800) × 10 = 3.125折 ≈ 3.13折
```

**改价校验**：

- 用户输入 ¥200 → 低于最低改价金额 ¥250 → 拒绝，显示错误
- 用户输入 ¥300 → 高于最低改价金额 ¥250 → 允许，确认改价

**改价对金额卡的影响**：

- 改价后，金额卡X仍然划扣¥100，金额卡Y仍然划扣¥150
- 改价只影响待收款：`待收款 = 改价金额 - ¥250`

#### 示例6：新规则 - 用户手动修改折扣后的金额卡计算

**场景**：home类型，原价¥200，数量1
- 用户先应用金额卡：卡1余额¥100（8折），卡2余额¥150（7折）
- 用户随后修改折扣为90%（`userDiscountOverride=1`）

**应用金额卡时（PRD规则）**：
| 计算步骤   | 卡1                              | 卡2                            |
| ------ | ------------------------------- | ----------------------------- |
| 折前价值   | ¥200.00                         | (继续)                          |
| 划扣金额   | min(200×0.8, 100)=¥80           | min(100×0.7, 150)=¥70         |
| 划扣原价值  | 80÷0.8=¥100                     | 70÷0.7=¥100                   |
| 剩余折前价值 | 200-100=¥100                    | 100-100=¥0                    |
| 剩余应付   | 0                               |                               |
| 折扣率    | round(150 ÷ 200 × 100) = 75%    |                               |

**用户修改折扣为90%后（新规则）**：
| 计算步骤   | 计算公式                 | 结果      |
| ------ | -------------------- | ------- |
| 商品折后价  | 200 × 1 × 0.9        | ¥180.00 |
| 卡1划扣金额 | min(180, 100)        | ¥100.00 |
| 剩余应付   | 180 - 100            | ¥80.00  |
| 卡2划扣金额 | min(80, 150)         | ¥80.00  |
| 剩余应付   | 80 - 80              | ¥0.00   |
| 实际支付   | 100 + 80 + 0         | ¥180.00 |
| 折扣率    | 180 ÷ 200 × 100      | 90%     |

**规则对比**：
| 规则类型 | 划扣基准 | 应付金额公式 |
| ---- | ---- | -------- |
| PRD规则 | 剩余折前价值 × 金额卡折扣率 | 折前价值 - Σ划扣原价值 |
| 新规则 | 商品折后价 | 商品折后价 - Σ划扣金额 |
