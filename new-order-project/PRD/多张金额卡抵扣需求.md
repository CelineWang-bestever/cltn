## create-order-pad.html 多张金额卡抵扣：应用逻辑说明与校验（评审用）

### 1. 目标与范围

本说明用于对齐并校验 `create-order-pad.html` 中“金额卡应用/抵扣”的核心业务逻辑与边界处理，确保文档描述与页面实现一致。

本说明覆盖（以当前实现为准）：
- 金额卡数据结构（商品行绑定、弹窗状态、明细行渲染）
- 选卡/替换/删除/防重复、默认划扣生成、0 金额自动移除
- 多卡折扣规则（取最小折扣）与折扣提示展示
- 划扣金额归一化（跨行、跨卡余额递减）、数量/折扣/折后价变化时的联动刷新策略
- 数字输入键盘的“确认后生效”原则在金额卡划扣金额上的落地（含合法性校验与错误提示阻断）
- 订单汇总的金额卡统计口径（金额卡支付/待收款）

不在本说明覆盖（当前原型未实现或仅模拟）：
- 金额卡有效性（过期/冻结/不可用）与后端校验的真实拦截
- 真实交易扣减、并发一致性、退款/撤销后的回滚与审计流水
- 支付页的多渠道实收/找零/组合支付的完整财务链路（本页仅维护“待收款/金额卡支付/共减”的展示口径）

### 2. 术语与数据口径

- 原价：商品行展示的单价（`.col-original`，单位元）
- 购买数量：商品行数量输入（服务/产品为 `data-field="buy"`；疗程为 `data-field="buyQty"`）
- 折扣：商品行折扣输入（百分比 0–100）。金额卡同步时为整数百分比；手动输入“折后价”反算时可能为 2 位小数
- 折后价（本页口径）：商品行折后总价（元），计算为 `原价 × 购买数量 × 折扣%`
- 金额卡折扣（折）：金额卡在不同商品类型下的折扣（例如 7.5 折）
- 划扣金额：某张金额卡在当前商品行上用于抵扣的金额（元）
- 行应付金额：`折后价 - Σ(该行所有金额卡划扣金额)`，并做 0 下限保护
- 订单金额卡支付：当前订单内所有商品行的金额卡划扣金额之和
- 订单待收款：`整单应收（含整单改价后的 displayFinal）- 订单金额卡支付`，并做 0 下限保护

### 3. 数据结构与渲染载体

#### 3.1 商品行与金额卡绑定（DOM 承载）

- `tr.item-row`：可应用金额卡的商品行（服务/产品/疗程）
- `data-moneycards`：JSON 数组，形如 `[{ "cardId": "mc_001", "amount": 100.00 }, ...]`
- 详情行：在 `item-row` 与其 `item-action-row` 之间插入 `tr.moneycard-detail-row`，用于展示已选金额卡明细与划扣金额输入
- 选择按钮：商品行操作区按钮 `[data-action="open-moneycard-modal"]`，文案随“已选 X 张金额卡”变化；当行应付=0 时置灰不可继续选卡

#### 3.2 金额卡列表与弹窗状态

- 金额卡卡池：`MONEY_CARD_OPTIONS`（包含 `id/name/balance/discounts`；余额为前端模拟的初始余额）
- 选卡弹窗状态：`moneyCardModalState`
  - `row`：当前操作的商品行
  - `mode/editIndex`：新增 or 替换（编辑）
  - `pendingId`：弹窗内当前选中的金额卡 id（支持“不使用金额卡”空值）

### 4. 业务规则 → 技术落地逻辑（与实现一致）

#### 4.1 选卡/替换/删除/防重复

**规则 1：新增金额卡（Add）**
- 当在弹窗中选择某金额卡并确认：
  - 若该行已存在同 cardId 的金额卡，则阻止重复添加（无更新）
  - 为新增卡生成默认划扣金额（见 4.3 规则 1），若默认划扣 ≤ 0，则直接关闭弹窗且不新增

**规则 2：替换金额卡（Replace / Edit）**
- 在明细行点击“替换”进入编辑模式：
  - 若选择“不使用金额卡”（空值）并确认：仅移除该 index 对应的金额卡
  - 若选择其他金额卡并确认：
    - 不允许替换成该行已存在的其他金额卡（防重复）
    - 替换仅更新 cardId，随后触发归一化（会按余额/剩余应付对划扣金额进行裁剪或自动补齐）
  - 编辑完成后会清除 `moneycardTouched`，使该行回到“系统自动分配”模式（见 4.3 规则 2）

**规则 3：删除金额卡**
- 明细行点击“删除”：移除对应金额卡并触发归一化；同时清除 `moneycardTouched`，使该行回到“系统自动分配”模式

#### 4.2 折扣取值与提示

**规则 1：折扣取最小（折）**
- 若商品行存在金额卡：取所有已选金额卡在该商品类型下的最小折扣（折），换算为百分比并写回折扣字段
- 商品类型与折扣字段映射：
  - 产品（product）→ `home`
  - 服务/疗程（service/course）→ `care`
  - 其他（默认）→ `item`

**规则 2：折扣提示**
- 当商品行存在任意金额卡时，在操作区常驻显示：
  - `使用已选金额卡最小折扣 X 折`
  - X 为当前最小折扣（折）
- 当商品行金额卡清空后：提示隐藏

**规则 3：清空后折扣回到 100%（有条件）**
- 当商品行金额卡数量变为 0 时：
  - 若该行未被标记 `userDiscountOverride=1`，折扣自动重置为 100%，并同步触发折后价重算
  - 若存在 `userDiscountOverride=1`（来自“用户确认修改折扣/折后价导致清卡”），则保留用户输入的折扣，不强制回 100%

#### 4.3 划扣金额生成与归一化（核心）

**规则 1：新增金额卡的默认划扣**
- 新增金额卡时，先用“加入该卡后的最小折扣”计算新的折后价（折后总价）：
  - `newLineTotal = 原价 × 数量 × (minFold×10)%`
- 默认划扣金额取“该行在新增前已划扣后的剩余金额”，并按该卡余额裁剪：
  - `remaining = max(0, newLineTotal - 已存在金额卡划扣之和)`
  - `defaultAmt = min(remaining, 该卡初始余额)`
- 若 `defaultAmt ≤ 0`：不新增该卡（直接关闭弹窗）

**规则 2：归一化的触发时机**
- 任一触发会执行 `normalizeAndRenderMoneyCards()` 并联动更新行应付与订单汇总：
  - 新增/替换/删除金额卡
  - 数量变化（buy/buyQty）：300ms 防抖触发
  - 折扣/折后价变化：raf/blur 定时触发（折后价 blur 后 80ms 内触发一次全量归一化）
  - 划扣金额变化：用户输入后确认（数字键盘 ✓ 或非键盘 blur）

**规则 3：跨行余额递减（同一张卡在订单内共享余额）**
- 归一化按固定行顺序遍历：服务区 → 产品区 → 疗程区（DOM 顺序）
- 对每张金额卡维护一个“运行中余额 runningBalance”：
  - 初始值为 `MONEY_CARD_OPTIONS.balance`
  - 每处理到一行的某张卡：先取该卡“处理前余额 before”，再扣减本行的划扣 amount 得到 after，写回 runningBalance

**规则 4：两种划扣分配模式（moneycardTouched）**
- 若 `row.dataset.moneycardTouched !== '1'`（未手动干预）：
  - 系统自动尽可能抵扣：`amount = min(该行剩余应付 itemRemaining, beforeBalance)`
- 若 `row.dataset.moneycardTouched === '1'`（用户手动调整过划扣金额）：
  - 以用户输入为基准，但会被归一化裁剪：
    - 不超过该行剩余应付
    - 不超过该卡处理前余额
  - 裁剪后若 amount ≤ 0，则该卡从该行移除

**规则 5：0 金额自动移除**
- 任意场景下出现 amount ≤ 0（例如用户输入 0、归一化后被裁剪到 0）：该金额卡从该行移除，并同步更新明细行与“已选 X 张金额卡”按钮文案

#### 4.4 用户手动修改折扣/折后价与金额卡的互斥（确认后才清卡）

**规则 1：手动修改折扣（仅在用户确认提交后触发）**
- 当行已应用金额卡时，用户通过数字键盘确认提交折扣，且“新值 ≠ 打开键盘时的原始值”：
  - 标记 `userDiscountOverride=1`
  - 清空该行所有金额卡（保留用户输入的折扣与折后价计算结果）

**规则 2：手动修改折后价（仅在用户确认提交后触发）**
- 当行已应用金额卡时，用户通过数字键盘确认提交折后价，且“新值 ≠ 打开键盘时的原始值”：
  - 标记 `userDiscountOverride=1`
  - 清空该行所有金额卡
  - 折扣按 `折扣% = 折后价 ÷ (原价 × 购买数量) × 100` 反算，保留 2 位小数

**规则 3：折后价下限校验（确认时拦截）**
- 当用户通过数字键盘确认折后价时：折后价不允许低于 `原价 × 购买数量`
- 校验失败：弹窗提示并阻断更新（不写回原字段）

#### 4.5 数字输入键盘在“金额卡划扣金额”上的合规交互原则

**规则 1：仅 ✓ 确认且校验通过才写回业务数据**
- 点击键盘 `✓` 前，所有输入均为弹窗 buffer，不影响行数据
- 点击键盘右上角关闭/点击遮罩关闭/点击键盘 `×` 清除：仅清空 buffer，不修改原字段值

**规则 2：划扣金额确认时的合法性校验（阻断更新）**
- 仅对 `.moneycard-deduct-input` 生效，校验顺序与口径如下：
  1) 划扣金额 ≤ 该商品折后价（行折后总价）
  2) 划扣金额 ≤ 该行剩余可划扣金额（折后价 - 其他金额卡已划扣之和）
  3) 划扣金额 ≤ 该金额卡当前剩余余额（按订单内共享余额的“处理前余额 beforeBalance”口径）
- 任一不通过：弹出明确错误提示并阻断更新，原字段值保持不变

#### 4.6 订单汇总统计口径（与金额卡抵扣一致）

- 订单金额卡支付：汇总所有 `tr.item-row[data-moneycards]` 内的 `amount` 之和（>0 的项）
- 待收款：`整单应收（含整单改价后的 displayFinal）- 订单金额卡支付`，并做 0 下限保护
- 仅当“订单金额卡支付 > 0”时，底部汇总区域展示独立字段「金额卡支付」

### 5. 边界与异常处理（实现已覆盖）

- 余额不足：系统会在归一化时按卡“处理前余额”裁剪，且后续行会使用扣减后的 runningBalance
- 行应付为 0：金额卡选择按钮置灰；若已选金额卡被裁剪为 0 会被移除
- 数量变化：300ms 防抖触发全量归一化，未 touched 的行会自动重新分配划扣金额以尽可能覆盖应付
- “不使用金额卡”：在编辑模式下可用于移除单张卡；在新增模式下仅关闭弹窗不做变更

### 6. 实现检查点（便于 Review）

- 商品行金额卡读写：`getRowMoneyCards(row)` / `setRowMoneyCards(row, apps)`
- 折扣同步与提示：`syncRowDiscountFromMoneyCards(row, apps)` / `setMinDiscountHint(row, minFold)`
- 清卡入口：`clearRowMoneyCards(row, options)`
- 归一化与明细行渲染：`normalizeAndRenderMoneyCards()`
- 选卡弹窗：`openMoneyCardModalForRow()` / `applyMoneyCardSelection()`
- 划扣金额键盘确认与校验：`commitNumKeypad()` 中 `.moneycard-deduct-input` 分支
- 订单汇总口径：`updateOrderSummary()` 中 `moneyCardTotal / displayPayable`

## 金额卡应用测试用例

以下测试用例分两类：
- A 类：与当前 `create-order-pad.html` 原型实现一致，可直接在页面复现与验收
- B 类：系统级应有场景（依赖后端/交易/退款链路），用于产品/研发联调验收；当前原型不提供真实数据支撑

### A 类（原型可执行）

#### 用例 A01：单行单卡足额抵扣

- 前置条件：商品行原价=100，数量=2；选择一张余额≥200 的金额卡（折扣任意）
- 操作步骤：打开选卡弹窗 → 选择该卡 → 点击确认
- 预期结果：折扣同步为该卡对应商品类型折扣的最小值；默认划扣金额=该行折后价；行应付=0；金额卡明细行显示余额为扣减后的值

#### 用例 A02：单行单卡余额不足部分抵扣

- 前置条件：商品行折后价=160；选择一张余额=100 的金额卡
- 操作步骤：选择该卡并确认
- 预期结果：默认划扣=100；行应付=60；明细行余额显示 0；“选择金额卡”按钮仍可继续选择（因为行应付>0）

#### 用例 A03：同一行多卡叠加，折扣取最小

- 前置条件：同一商品行可选择两张折扣分别为 9 折与 7.5 折的金额卡
- 操作步骤：先添加 9 折卡，再添加 7.5 折卡
- 预期结果：折扣字段写入 75（整数百分比）；提示显示“最小折扣 7.5 折”；折后价按 75% 重算；两张卡的划扣金额按“剩余应付 + 余额”规则生成/裁剪

#### 用例 A04：替换金额卡（Replace）后重新归一化

- 前置条件：某商品行已应用 2 张金额卡
- 操作步骤：在其中一张卡上点击“替换” → 选择另一张未使用的卡 → 确认
- 预期结果：该 index 的 cardId 被替换；不出现重复卡；该行恢复为“系统自动分配”（touched 清除），归一化后划扣金额与余额展示合理

#### 用例 A05：替换为“不使用金额卡”（删除单张）

- 前置条件：某商品行已应用 2 张金额卡
- 操作步骤：在其中一张卡上点击“替换” → 选择“不使用金额卡” → 确认
- 预期结果：仅该卡被移除；折扣/提示/折后价/行应付按剩余卡重新计算；若仍有金额卡则提示继续显示

#### 用例 A06：删除最后一张卡触发重置

- 前置条件：某商品行仅应用 1 张金额卡，且该行未被标记 userDiscountOverride
- 操作步骤：点击该卡“删除”
- 预期结果：金额卡明细行移除；提示消失；折扣重置为 100；折后价按 100% 重算；行应付=折后价

#### 用例 A07：划扣金额键盘输入，点击 × 清除不生效

- 前置条件：某商品行已应用金额卡，明细行存在划扣金额输入框且当前值为非 0
- 操作步骤：点击划扣金额输入框唤起数字键盘 → 输入新数值但不点 ✓ → 点击键盘 × 清除
- 预期结果：仅键盘展示区清空；原输入框值保持不变；行应付/订单汇总不发生变化

#### 用例 A08：划扣金额键盘输入，点击关闭不生效

- 前置条件：同 A07
- 操作步骤：输入新数值但不点 ✓ → 点击键盘右上角关闭/点击遮罩关闭
- 预期结果：原输入框值保持不变；行应付/订单汇总不发生变化

#### 用例 A09：划扣金额输入超出折后价，确认拦截

- 前置条件：该行折后价=160；当前已划扣为 100
- 操作步骤：对某张卡的划扣金额输入 200 → 点击 ✓
- 预期结果：弹出错误提示“划扣金额不可超过该商品折后价”；不更新原值

#### 用例 A10：划扣金额输入超出剩余可划扣金额，确认拦截

- 前置条件：该行折后价=160；另一张卡已划扣=120
- 操作步骤：对当前卡输入 60 → 点击 ✓
- 预期结果：弹出错误提示“不可超过该商品剩余可划扣金额(¥40.00)”；不更新原值

#### 用例 A11：划扣金额输入超出该卡当前剩余余额，确认拦截

- 前置条件：同一张金额卡在前序行已抵扣导致“处理前余额 beforeBalance=30”
- 操作步骤：在当前行对该卡输入 50 → 点击 ✓
- 预期结果：弹出错误提示“不可超过该金额卡剩余余额(¥30.00)”；不更新原值

#### 用例 A12：数量变化触发自动重算（未 touched）

- 前置条件：某行已选金额卡但未手动改过划扣金额（moneycardTouched 不存在）；当前行应付>0
- 操作步骤：修改数量（buy/buyQty）使折后价变化
- 预期结果：300ms 内触发归一化；该行金额卡划扣金额自动按“尽可能覆盖应付 + 余额”重新分配；余额展示随之变化；订单汇总同步更新

#### 用例 A13：数量变化不覆盖用户手动输入（touched）

- 前置条件：某行已通过 ✓ 确认修改过划扣金额（moneycardTouched=1）
- 操作步骤：修改数量（buy/buyQty）
- 预期结果：归一化仍会裁剪超限（不超过应付/余额），但不会“自动补齐为最大”；用户输入的意图被保留

#### 用例 A14：手动改折扣/折后价（确认后）触发清卡

- 前置条件：某商品行已应用金额卡；记住折扣/折后价的原值
- 操作步骤：点击折扣或折后价打开数字键盘 → 输入与原值不同的新值 → 点击 ✓
- 预期结果：该行金额卡被清空；折扣提示消失；折后价/折扣按新值计算；订单汇总中金额卡支付随之减少

### B 类（系统级应有，依赖后端/交易链路）

#### 用例 B01：过期金额卡拦截

- 前置条件：后端下发金额卡状态=已过期，且该卡在可选列表中可被识别为不可用
- 操作步骤：尝试选择该过期卡并确认
- 预期结果：弹出明确错误提示并阻断选择；不写入 data-moneycards；折扣/应付不改变

#### 用例 B02：无效/冻结金额卡拦截

- 前置条件：后端下发金额卡状态=冻结/无效
- 操作步骤：选择并确认
- 预期结果：同 B01

#### 用例 B03：金额卡与其他优惠叠加规则校验

- 前置条件：订单存在其他优惠（券/满减/整单改价等），且产品定义了与金额卡的叠加/互斥规则
- 操作步骤：先应用其他优惠，再应用金额卡（或反向顺序），完成结算
- 预期结果：折扣/优惠的优先级与互斥策略符合财务口径；金额卡支付金额与待收款一致可对账

#### 用例 B04：订单退款后金额卡退回（全退/部分退）

- 前置条件：订单已支付完成且金额卡已发生真实扣减；存在退款接口与卡余额回滚能力
- 操作步骤：发起全额退款 / 部分退款
- 预期结果：金额卡扣减按退款金额按规则退回；产生可审计的资金流水；同一订单多卡抵扣时按规则回滚到对应卡
